# Inspire RH56E2 grasp — заметки по интеграции

Этот каталог — drop-in замена старого `g1_manip_node.py` из репо
`kuzmich-coffee-bot` (`ros2_g1_cup_solution/scripts/g1_manip_node.py`),
переписанная под прямое ModbusTCP-управление кистью Inspire RH56E2 (Gen4),
без Unitree SDK.

## Что в каталоге

| Файл | Назначение |
|------|------------|
| `inspire_modbus.py` | Zero-dependency ModbusTCP клиент (FC03 / FC16 на raw sockets). |
| `inspire_e2_hand.py` | Класс `InspireE2Hand` — Python-порт C++ контроллера. Реализует `get_hand_state()` / `set_command()` / `get_touch_state()` и хелперы для grasp-логики. |
| `g1_manip_node.py` | ROS2-нода `g1_manip_node` с сервисами `/g1_manip/grasp_cup` и `/g1_manip/release_cup` (drop-in совместимый со старым интерфейсом). |
| `inspire_hand_probe.py` | CLI для **обязательной** предстартовой верификации регистров. |
| `__init__.py` | Пустой, для удобства импорта. |

## Карта регистров (статус верификации)

| Регистр | Адрес | Op | Длина | Статус | Источник |
|---------|-------|----|-------|--------|----------|
| `HAND_ID`         | 1000  | R  | 1  | ✅ Подтверждён чтением с живой руки | ТЗ |
| `ANGLE_ACT`       | 1546  | R  | 6  | ✅ Подтверждён чтением (значения `[45,2,62,259,2,1000]`) | ТЗ + C++ |
| `ANGLE_SET`       | **1009** | W | 6 | ⚠️ **Документированное значение для Inspire RH56 series — обязательно верифицировать probe-скриптом** | Документация Inspire |
| `TACTILE_BASE`    | 3000  | R  | 17 | ⚠️ **Считаем 17 пиков подряд начиная с 3000**. ТЗ говорит диапазон 3000–4900 (там сырые данные), но пики по 17 пэдам скорее всего лежат подряд в начале. Если probe показывает все нули на касании — нужно искать правильный offset в `inspire-api` или C++ контроллере. | ТЗ + предположение |
| `TOUCH_FULL_SCALE`| —     | —  | —  | ✅ 4095 | ТЗ |

## Что нужно сделать на железе перед первым запуском

Порядок строго обязательный (ТЗ прямо запрещает гадать адрес `ANGLE_SET`).

### 1. Прочитать memory-файл другой сессии

```
~/.claude/projects/-home-unitree-teleop/memory/inspire-rh56e2-hand-integration.md
```

Там могут быть детали, которых нет в ТЗ. Особенно — точный адрес `ANGLE_SET`
и layout тактильных регистров.

### 2. Прогнать детектор

```bash
python3 thirdparty/inspire_hand_detect.py --ip 192.168.123.210
```

Должен показать, что левая рука жива. Если правая ещё не восстановлена —
ожидаемо, что её нет на связи.

### 3. Прогнать probe (ключевой шаг)

```bash
# Сначала — без motion, только чтение + no-op тест:
python3 inspire_hand_probe.py --ip 192.168.123.210
```

Скрипт:
- читает `HAND_ID`, `ANGLE_ACT`, 17 тактильных пэдов;
- делает **безопасный no-op write**: читает текущий `ANGLE_ACT` и пишет те
  же значения в кандидат `ANGLE_SET` (рег 1009). Рука не должна двигаться;
- если приходит Modbus exception `illegal data address` — кандидат неверный.
  Скрипт подскажет, какие ещё адреса попробовать (1003, 1007, 1011, 1015).

Если no-op тест **провалился**, нужно найти правильный адрес в:
- `thirdparty/inspire-api/` — Python-реализация протокола от Inspire;
- `src/cpp/hands/inspire_e2/inspire_e2_hand.cpp` — уже проверенный на железе
  C++ контроллер. **Это самый надёжный референс** — он уже гонялся на живой
  руке и читал реальное состояние `[45,2,62,259,2,1000]`.

После того как найден правильный адрес — поправить константу в
`inspire_e2_hand.py`:

```python
ANGLE_SET_REG = 1009   # <- заменить на найденное значение
```

### 4. Опционально — move-test (только с пустой рукой!)

```bash
python3 inspire_hand_probe.py --ip 192.168.123.210 --move-test
```

Делает маленький цикл q=0.9 → 0.7 → 0.9 (10% от диапазона). Подтверждает,
что `ANGLE_SET` действительно двигает пальцы. **Убедитесь, что в руке ничего
нет** — ни чашки, ни пальцев оператора.

### 5. Старт ROS2-ноды в «read-only» режиме

В `g1_manip_node.py` параметр `enable_motion` по умолчанию `False`. Это
значит: нода поднимется, ответит на вызовы сервисов, но **не будет писать в
`ANGLE_SET`**. Так можно проверить, что нода видит руку и тактильные пэды,
без риска что-то сломать.

```bash
ros2 run g1_manip_solution g1_manip_node
# в другом терминале:
ros2 service call /g1_manip/grasp_cup std_srvs/Trigger
# -> response.success=False, message="MOTION_DISABLED" — ожидаемо
```

### 6. Включить motion

```bash
ros2 run g1_manip_solution g1_manip_node --ros-args -p enable_motion:=true
```

## Логика захвата (по ТЗ, адаптирована под тактильность)

В `_grasp_one()`:

1. Открыть кисть полностью (`set_uniform(1.0)`), подождать `open_settle_time`.
2. Снять baseline тактильного состояния.
3. Цикл закрытия:
   - уменьшить `q` на `grasp_step` (0.05 по умолчанию);
   - записать `set_uniform(q)` в `ANGLE_SET`;
   - прочитать `get_touch_state()`;
   - **spike-check**: если дельта любого пада между двумя семплами больше
     `spike_threshold` (0.30) — немедленно приоткрыть на `back_off_step`
     (0.20) и вернуть `SPIKE_BACKOFF` (или продолжить, если
     `fail_on_spike=false`);
   - **hold-check**: если `peak_fingertip_pressure >= hold_threshold` (0.05)
     **и** хотя бы `min_contact_pads` (3) пэдов выше `contact_threshold`
     (0.03) — захват зафиксирован, вернуть `OK`;
   - иначе — следующий шаг;
   - кап `max_close_steps` (40) или `q <= 0` → `NO_CONTACT`.
4. `release_cup` — `set_uniform(1.0)`, подождать `release_settle_time`.

Все параметры — ROS2-параметры ноды, тюнятся через `--ros-args -p ...`.

## Что НЕ сделано (намеренно, по ТЗ)

- **Не интегрируется в `~/teleop/`** — это чужая лицензированная система под
  телеоперацию. ТЗ прямо запрещает.
- **Правая рука** — поддерживается на уровне класса, но по умолчанию
  `target_hand=left`. На момент написания ТЗ правая рука физически на связи
  нет (заводской IP `.11.210`, попытка перепрошивки не удалась). Когда
  восстановят — достаточно прописать её IP в `right_hand_ip` и переключить
  `target_hand`.
- **Старая force_sensor.py логика «граммы»** — выкинута полностью. ТЗ прямо
  говорит, что тактильные пэды дают более честный сигнал «есть контакт /
  нет», градуировка по граммам не нужна.

## Допущения (явные, чтобы потом не было сюрпризов)

1. **`ANGLE_SET = 1009`** — документированное значение для Inspire RH56
   series (включая RH56E2 Gen4). Протокол командных регистров у Inspire
   совпадает между RH56DFTP и RH56E2, поэтому с высокой вероятностью
   1009 верно. **Но ТЗ прямо запрещает гадать — поэтому probe-скрипт
   делает no-op тест перед motion.**
2. **Тактильные пики лежат подряд в 17 регистрах начиная с 3000.** ТЗ говорит
   «диапазон 3000–4900» — это сырые данные (включая координаты центров
   давления, события и т.п.). Для grasp-логики нам нужны только пики, и они
   с большой вероятностью лежат в начале диапазона. Если probe показывает
   нули при реальном касании — нужно найти правильный offset и поправить
   `TACTILE_BASE_REG`.
3. **Порядок суставов `[pinky, ring, middle, index, thumbBend, thumbRot]`**
   взят из C++ контроллера, где он явно помечен как
   «ПРЕДПОЛОЖЕНИЕ, не подтверждено на железе». Для grasp-логики это не
   критично — мы пишем одно и то же значение во все 6 суставов (`set_uniform`).
   Станет критично, если захочется индивидуальный контроль пальцев — тогда
   сделать gentle one-finger nudge и посмотреть, какой регистр `ANGLE_ACT`
   меняется.
4. **Порядок пэдов** — `[pinky/ring/middle/index × (top,tip,base), thumb ×
   (top,tip,mid,base), palm]`. Тоже из C++ контроллера. Для grasp-логики
   важны только fingertip-индексы (1, 4, 7, 10, 13) — если порядок на
   железе окажется другим, `TIP_PAD_INDICES` надо будет поправить.
5. **`touch_full_scale = 4095`** — из ТЗ. Реальные значения при касании
   доходят только до ~0.21 от шкалы — это норма (ТЗ упоминает), пороги
   `hold_threshold=0.05` и `contact_threshold=0.03` подобраны под это.

## Известные риски

- **Если `ANGLE_SET` ≠ 1009** — первый реальный write может увести кисть в
  неизвестное состояние. Поэтому `enable_motion=False` по умолчанию и
  обязательный probe-шаг.
- **Если тактильный layout другой** — grasp-логика будет думать, что
  контакта нет, и закроет кисть до конца. На пустой руке это безопасно
  (просто закроется), но если в руке уже что-то есть — может смять. Поэтому
  `max_close_steps` ограничивает количество шагов.
- **Spike detection** — чувствителен к шагу дискретизации. При
  `grasp_step=0.05` и `grasp_step_delay=0.08` разница между двумя семплами
  может быть значительной даже при нормальном захвате. Если будут ложные
  срабатывания — увеличить `spike_threshold` или уменьшить `grasp_step`.

## Интеграция в репо `kuzmich-coffee-bot`

1. Скопировать 4 файла в `ros2_g1_cup_solution/scripts/`:
   ```
   inspire_modbus.py
   inspire_e2_hand.py
   g1_manip_node.py          # затирает старую версию
   inspire_hand_probe.py
   ```
2. В `setup.py` / `package.xml` добавить `g1_manip_node` как entry-point
   (если уже было — не нужно трогать, имя ноды сохранилось).
3. Убедиться, что `setup.py` копирует все 4 `.py` файла (не только
   `g1_manip_node.py`).
4. `colcon build`, `source install/setup.bash`.
5. Прогнать probe на живой руке (см. выше).
6. Стартануть ноду в read-only, потом с `enable_motion:=true`.
